# 面試應答腳本：AI 驅動的自動化數位工廠 (STAR Method)

**情境 (Situation)**：
「在之前的專案中，我們有一個基於 Python 開發的工廠模擬核心，以及一個用 React 寫的前端儀表板。當時面臨的最大痛點是，這兩者是完全脫節的——前端只是一個靜態的 UI 殼，而後端的模擬邏輯雖然在跑，但缺乏『大腦』。當訂單大量湧入或機器過熱時，系統無法自動反應，導致生產效率低落，且使用者無法直觀地監控或控制產線，這讓整套系統停留在了『展示品』而非『實用工具』的階段。」

**任務 (Task)**：
「因此，我主動發起了這項專案的重構計畫。我的目標很明確：**打造一個真正具備『自主決策能力』的數位孿生工廠 (Digital Twin)**。這不僅需要打通前後端的即時數據流，更關鍵的是要整合 **LLM (大型語言模型)** 作為工廠的中控大腦，讓它能像真人廠長一樣，24小時監控數十個數據點，並在毫秒級的時間內做出決策（如自動降溫或負載平衡）。」

**行動 (Action)**：
「為了達成這個目標，我主要做了三層架構的技術突破：

1.  **即時通訊層 (Real-time Bridge)**：
    我放棄了傳統的輪詢 (Polling) 方式，改用 **WebSocket** 架構了一個雙向橋接器。這實現了 60Hz 的高頻數據更新，讓前端儀表板能『零時差』地反映後端每一台機器的震動與溫度，解決了數據不同步的問題。

2.  **AI 決策核心 (AI Integration)**：
    這是最具挑戰的部分。我使用了本地部署的 **Ollama** 模型，並設計了一套嚴謹的 **Prompt Engineering** 機制，強制 AI 輸出標準化的 JSON 指令。為了防止 AI『幻覺』導致工廠失控，我實作了 **Safety Limits (安全邊界)** 和 **Timeout 機制**——如果 AI 在 10 秒內沒反應，或者下達了超出物理極限的轉速指令，系統會自動攔截並觸發降級保護 (Fail-safe)，確保工廠運作的絕對安全。

3.  **策略優化 (Strategic Logic)**：
    我觀察到單純的自動化還不夠，還需要『智慧』。因此我編寫了高階策略規則，教導 AI 執行 **『漸進式控制』**（例如過熱時慢慢降速，而不是直接停機造成產能損失），並實現了 **『跨產線負載平衡』**。也就是說，當 A 產線爆單時，閒置的 C 產線會自動接手支援。為了讓這個邏輯可視化，我還重寫了前端的訂單追蹤邏輯，讓跨線生產的狀態能準確顯示在 UI 上。」

**結果 (Result)**：
「最終，這個專案將原本單調的模擬器，轉變成了一個全自動化的智慧工廠系統。
在量化成果上，我們成功透過負載平衡機制，**顯著提升了訂單處理的吞吐量** (原本閒置的產線現在能主動分擔積壓訂單)，並消除了所有因 AI 延遲導致的系統當機。
在質化體驗上，這套系統現在支援自然語言交互——我可以像跟同事說話一樣，在對話框輸入『幫我降溫』，工廠就會自動執行一連串複雜的降速操作。這證明了我有能力將生成式 AI 從『聊天機器人』落地到『複雜系統控制』的實際應用場景中。」
